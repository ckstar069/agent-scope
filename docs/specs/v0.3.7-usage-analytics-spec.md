# AgentScope v0.3.7 Usage Analytics Spec

## 1. 背景与目标

### 1.1 背景

AgentScope v0.3.6 已完成以下核心能力收口：

- Agent 监控首屏快照与会话管理加载性能优化
- 记忆资产 dashboard 合并扫描、异步化与刷新去重
- Token 速率计算语义修正：不再使用 `active_tokens / session.elapsed()` 伪全程均速
- 窗口速率采用 1 分钟、5 分钟窗口内 token delta
- 速率 UI 明确为估算性质
- 会话预览去除 `last-prompt` 造成的伪重复
- 默认折叠工具调用，避免工具噪音淹没用户/助手正文
- 关键指标增加轻量 Tooltip
- 速率统计按钮合并为单组：5 分钟平均、1 分钟平均、监控期平均、最近 2 秒

在 v0.3.6 发布后的实际使用中发现，当前 Token 速率仍会出现"跳变/飘忽"的观感。经排查，这主要不是 AgentScope 计算 bug，而是**数据源机制**决定的：

- 当前通过 `abtop-collector` 读取 Claude Code 本地 session/transcript 文件
- Token usage 通常来自 Claude Code JSONL 中 assistant 消息的 `usage` 字段
- 该字段往往在**回复完成后才写入**
- 因此流式输出过程中，本地 usage 不一定实时增长
- 速率本质是基于已写入 usage 的采样估算

### 1.2 v0.3.7 目标

v0.3.7 定位为 **"Usage Analytics Foundation"**，目标是补齐主流工具（ccusage / Claude-Code-Usage-Monitor）的 usage analytics 基础能力：

- 数据源兼容增强（多目录发现）
- Usage source / confidence 标识（可信度与透明度）
- 最小历史 usage 报表（今日/近 7 天/按项目/按模型/按会话）
- JSON 导出
- ccusage 对账诊断

**不是**要做 API Proxy，也不是要替代 ccusage，而是在桌面端提供与 AgentScope 项目监控体系融合的 usage 分析能力。

### 1.3 v0.3.7 明确不做

| 不做项 | 说明 |
|--------|------|
| 实时 API Proxy | 不拦截 Claude Code 网络请求 |
| 流式 token 精确统计 | 仍然基于本地 JSONL 文件，不增加 streaming 计数器 |
| 成本账单级精确核算 | 不做完整价格表和计费系统 |
| 替代 ccusage | ccusage 仍是独立 CLI 工具，AgentScope 只提供交叉校验 |
| 重构 v0.3.6 Agent Monitor 采集主链路 | Agent Collector 的 2 秒轮询和实时面板保持不变 |

## 2. P0 范围

### 2.1 P0-1：Usage 数据源兼容增强

补齐 Claude Code 数据目录发现能力：

- 支持 `~/.claude`（macOS/Linux 默认）
- 支持 `~/.config/claude`（XDG 规范路径）
- 支持 `CLAUDE_CONFIG_DIR` 环境变量（单目录或多目录逗号分隔）
- 支持多个目录合并扫描，自动去重
- UI 或诊断接口中显示已识别数据目录列表
- 不可读目录给出诊断提示，不导致崩溃

### 2.2 P0-2：Usage Source / Confidence 标识

让用户理解数据的来源和可信度：

- 标明数据来自 **"Claude Code 本地 JSONL usage 记录"**
- 标明实时性为 **"延迟更新"**（非 streaming 实时）
- 标明最终总量较可靠，实时速率为估算
- 显示最近一次 usage 更新时间
- 显示最近一次 JSONL 写入时间或类似指标
- active 会话 token 计数未刷新时显示 **"等待计数更新"**

### 2.3 P0-3：最小 Usage Analytics 页面或面板

提供基本历史用量视图：

- **时间范围**：今日、最近 7 天
- **汇总指标**：Total Tokens、Input、Output、Cache Read、Cache Create、Session Count
- **分组表格**：
  - 按项目聚合
  - 按模型聚合
  - 按会话聚合
- **分项统计**：input_tokens、output_tokens、cache_read_input_tokens、cache_creation_input_tokens
- **JSON 导出**：导出当前视图数据为 JSON 文件

### 2.4 P0-4：ccusage 对账诊断脚本或命令

验证 AgentScope usage 解析结果与主流工具的一致性：

- 用于对比 AgentScope 解析结果与 ccusage 输出
- 首期为开发/诊断脚本，不一定进入 UI
- 输出差异摘要（session 数量、token totals、缺失 session 等）

## 3. 非目标范围（v0.3.7 暂不做）

以下功能明确列入 **v0.4+ / v0.5+** 路线：

| 功能 | 暂缓原因 |
|------|---------|
| 5 小时额度窗口完整预测 | 需要更多数据积累，且 UI 设计较复杂 |
| 成本估算和价格表管理 | 价格表维护成本高，且各 provider 差异大 |
| P90 burn rate | 统计层面较复杂，需先稳定基础聚合 |
| 多模型价格配置 | 需要完整价格表设计 |
| statusline / HTTP endpoint | 属于外部消费出口，非桌面端核心 |
| API Proxy | 侵入式改造，不在 v0.3.7 范围 |
| 流式 token 估算 | 技术复杂度高，收益不明确 |
| 图表高级交互（drill-down、时间轴缩放） | 先保证基础表格可用 |
| 历史数据库持久化大重构 | 当前基于文件扫描已足够，除非性能瓶颈 |

## 4. 数据源设计

### 4.1 数据目录发现规则

```rust
// 伪代码：目录发现优先级
fn discover_claude_config_dirs() -> Vec<PathBuf> {
    let mut dirs = Vec::new();

    // 1. CLAUDE_CONFIG_DIR 环境变量（最高优先级）
    if let Ok(env_dirs) = std::env::var("CLAUDE_CONFIG_DIR") {
        for dir in env_dirs.split(',') {
            dirs.push(PathBuf::from(dir.trim()));
        }
    }

    // 2. 默认目录（按平台）
    #[cfg(target_os = "macos")]
    dirs.push(home_dir().join(".claude"));

    #[cfg(target_os = "linux")]
    {
        dirs.push(home_dir().join(".config/claude"));
        dirs.push(home_dir().join(".claude")); // 兼容旧版
    }

    #[cfg(target_os = "windows")]
    dirs.push(roaming_app_data().join("Claude"));

    // 3. 去重 + 过滤不可读
    dirs.into_iter()
        .unique()
        .filter(|d| d.exists() && d.is_dir())
        .collect()
}
```

### 4.2 默认目录（按平台）

| 平台 | 默认目录 |
|------|---------|
| macOS | `~/.claude` |
| Linux | `~/.config/claude`、`~/.claude`（兼容旧版） |
| Windows | `%APPDATA%/Claude` |

### 4.3 CLAUDE_CONFIG_DIR 处理

- 支持逗号分隔的多目录，如 `/path/a,/path/b`
- 每个目录独立扫描，结果合并
- 环境变量设置后**覆盖**默认目录（不叠加）
- 如需叠加，需用户显式在环境变量中包含默认目录

### 4.4 多目录去重

- 基于 **canonical path** 去重（resolve symlink）
- 相同 canonical path 的目录只扫描一次
- 不同目录下相同 session_id 的 usage 记录需要去重（见 4.5）

### 4.5 session_id 去重

- 以 `session_id` + `config_dir` 为复合 key
- 同一 session_id 出现在多个目录时，保留**文件修改时间更新**的版本
- 记录去重日志（debug 级别）

### 4.6 project path 映射

- JSONL 文件路径通常包含项目目录信息，如 `~/.claude/projects/{project-name}/usage.jsonl`
- 提取 `{project-name}` 作为默认项目名
- 若无法提取，标记为 `"unknown"`
- AgentScope 的 `ProjectRegistry` 中的项目路径与 usage 项目路径进行前缀匹配，建立映射
- 未匹配到的 usage 记录归入 `"未关联项目"`

### 4.7 JSONL 文件扫描策略

```
{config_dir}/
├── sessions/
│   └── {session-id}.json          # 会话元数据
├── projects/
│   └── {project-name}/
│       └── usage.jsonl            # 项目级 usage 记录
└── usage.jsonl                    # 全局 usage 记录（如果存在）
```

- 扫描范围：`{config_dir}/projects/**/usage.jsonl` + `{config_dir}/usage.jsonl`
- 每行一个 JSON 对象
- 只读取包含 `usage` 字段的 `assistant` 类型消息
- 忽略解析失败的行（记录日志），不中断扫描

### 4.8 增量扫描与全量扫描的边界

| 场景 | 策略 |
|------|------|
| 首次扫描 | 全量扫描所有目录的所有 JSONL 文件 |
| 定时刷新 | 增量扫描：基于文件 mtime，只读取 mtime > last_scan_at 的文件 |
| 手动刷新 | 全量扫描（用户点击刷新按钮） |
| 文件追加 | 基于文件大小/offset，只读取新增行 |

- 记录每个文件的 `last_offset` 和 `last_mtime`
- 增量扫描时，如果文件被截断（大小 < last_offset），回退到全量读取

### 4.9 错误处理

| 错误场景 | 处理方式 |
|---------|---------|
| 目录不存在 | 跳过，记录 debug 日志 |
| 目录不可读（权限不足） | 标记为 `unreadable`，返回给前端展示 |
| JSONL 文件损坏（单行无效 JSON） | 跳过该行，记录 warn 日志，继续扫描 |
| JSONL 文件不可读 | 跳过该文件，记录 warn 日志 |
| usage 字段缺失或格式异常 | 跳过该记录，记录 warn 日志 |
| 超大 JSONL 文件（> 100MB） | 分段读取，避免一次性加载到内存 |

### 4.10 不可读目录展示

- 在数据源状态卡片中列出所有已识别目录
- 可读的目录显示为绿色/正常状态
- 不可读的目录显示为红色/警告状态，附带错误原因（如"权限不足"）
- 空目录显示为黄色/提示状态

### 4.11 与 abtop-collector 的关系

| 组件 | 用途 | 是否复用 |
|------|------|---------|
| `abtop-collector` | 实时 Agent 监控（2 秒轮询进程 + 解析当前活跃 session） | **保持独立** |
| Usage Analytics Scanner | 历史 usage 报表（扫描 JSONL 文件，聚合统计） | **新增独立模块** |

**设计原则**：

- `abtop-collector` 专注**实时监控**，不做历史聚合
- Usage Analytics Scanner 专注**历史统计**，不依赖进程状态
- 两者共用 **JSONL Parser**（建议抽取为 `shared/usage_parser.rs`）
- Usage Analytics Scanner 可以复用 `abtop-collector` 的目录发现逻辑（如果合适），但独立维护自己的扫描状态

## 5. 数据模型设计

### 5.1 UsageRecord（单条 usage 记录）

```rust
struct UsageRecord {
    // 数据源标识
    source: UsageSource,           // "claude_jsonl"
    config_dir: PathBuf,           // 数据来源目录

    // 项目关联
    project_path: Option<String>,  // 项目路径（从 JSONL 路径推断）
    project_name: Option<String>,  // 项目名称（与 ProjectRegistry 匹配后填充）

    // 会话标识
    session_id: String,            // 会话 ID

    // 模型信息
    model: Option<String>,         // 模型名称（如 "claude-sonnet-4-6"）

    // 时间
    timestamp: DateTime<Utc>,      // usage 记录时间（从 JSONL 中的 message timestamp）

    // Token 统计
    input_tokens: u64,
    output_tokens: u64,
    cache_read_tokens: u64,
    cache_create_tokens: u64,
    total_tokens: u64,             // input + output + cache_read + cache_create

    // 溯源
    raw_file_path: PathBuf,        // 原始 JSONL 文件路径
    line_no: u64,                  // 行号（用于调试和对账）
}
```

**P0 必需字段**：全部字段均为 P0 必需。

### 5.2 UsageAggregate（聚合结果）

```rust
struct UsageAggregate {
    // 聚合维度
    time_range: TimeRange,         // 时间范围（今日/近7天/自定义）
    group_by: GroupBy,             // 分组维度（project/model/session）

    // 汇总指标
    input_tokens: u64,
    output_tokens: u64,
    cache_read_tokens: u64,
    cache_create_tokens: u64,
    total_tokens: u64,

    // 计数
    session_count: usize,
    project_count: usize,
    model_count: usize,

    // 明细
    groups: Vec<UsageGroup>,       // 各分组的明细
}

struct UsageGroup {
    group_key: String,             // 分组键（项目名/模型名/会话ID）
    group_label: String,           // 展示标签
    input_tokens: u64,
    output_tokens: u64,
    cache_read_tokens: u64,
    cache_create_tokens: u64,
    total_tokens: u64,
    session_count: usize,
    first_seen: DateTime<Utc>,
    last_seen: DateTime<Utc>,
}
```

**P0 必需字段**：全部字段均为 P0 必需。后续可扩展：

- `cost_usd: Option<f64>`（v0.4+）
- `burn_rate: Option<f64>`（v0.4+）
- `p50_tokens: u64`、`p90_tokens: u64`（v0.4+）

### 5.3 UsageSourceStatus（数据源状态）

```rust
struct UsageSourceStatus {
    source_type: String,           // "claude_jsonl"

    // 目录状态
    config_dirs: Vec<PathBuf>,     // 所有已识别目录
    readable_dirs: Vec<PathBuf>,   // 可读目录
    unreadable_dirs: Vec<UnreadableDir>,

    // 时间
    last_scan_at: Option<DateTime<Utc>>,     // 最近一次扫描时间
    last_usage_at: Option<DateTime<Utc>>,    // 最近一次发现 usage 记录的时间

    // 可信度
    confidence: ConfidenceLevel,   // "high" / "medium" / "low"
    realtime_level: RealtimeLevel, // "delayed" / "near_realtime" / "realtime"

    // 说明
    notes: Vec<String>,            // 用户可见的提示文本
}

struct UnreadableDir {
    path: PathBuf,
    reason: String,                // 错误原因
}

enum ConfidenceLevel {
    High,    // 最终总量较可靠
    Medium,  // 部分数据可能缺失
    Low,     // 数据源不可读或为空
}

enum RealtimeLevel {
    Delayed,         // 基于文件写入，延迟秒级到分钟级
    NearRealtime,    // 未来可能的改进
    Realtime,        // 未来 API Proxy 才可能实现
}
```

**P0 必需字段**：全部字段均为 P0 必需。

### 5.4 内存与缓存策略

- UsageRecord 在内存中以 `Vec<UsageRecord>` 形式缓存
- 数据量预估：1000 条记录 ≈ 500KB，可安全缓存在内存
- 如果数据量过大（> 10MB），考虑按日期分片缓存
- 聚合结果不持久化到磁盘，每次按需计算（数据量小时足够快）
- 未来可考虑 SQLite 持久化（v0.4+）

## 6. API / Tauri Command 设计

### 6.1 `get_usage_source_status`

- **用途**：获取当前 usage 数据源的状态和诊断信息
- **输入**：无
- **输出**：`UsageSourceStatus`
- **是否 spawn_blocking**：否（纯内存操作）
- **是否扫描文件系统**：只检查目录存在性和可读性，不扫描文件内容
- **是否适合缓存**：适合（可缓存 5 秒）
- **错误返回**：Tauri 标准错误（字符串）

### 6.2 `scan_usage_data`

- **用途**：触发 usage 数据扫描（全量或增量）
- **输入**：
  ```rust
  struct ScanUsageDataRequest {
      full_scan: bool,             // true = 全量扫描，false = 增量扫描
  }
  ```
- **输出**：
  ```rust
  struct ScanUsageDataResponse {
      scanned_dirs: usize,
      scanned_files: usize,
      scanned_lines: usize,
      new_records: usize,
      total_records: usize,
      duration_ms: u64,
      errors: Vec<String>,         // 扫描过程中的非致命错误
  }
  ```
- **是否 spawn_blocking**：**是**（文件 I/O 操作）
- **是否扫描文件系统**：是（核心操作）
- **是否适合缓存**：不适用（这是写操作）
- **错误返回**：致命错误（如权限不足导致完全无法扫描）返回 Tauri 错误；非致命错误包含在响应中

### 6.3 `get_usage_analytics`

- **用途**：获取 usage 聚合统计
- **输入**：
  ```rust
  struct GetUsageAnalyticsRequest {
      time_range: TimeRange,       // "today" | "last_7_days"
      group_by: GroupBy,           // "project" | "model" | "session"
  }

  enum TimeRange {
      Today,
      Last7Days,
      // Custom { start: DateTime<Utc>, end: DateTime<Utc> }, // P1+
  }

  enum GroupBy {
      Project,
      Model,
      Session,
  }
  ```
- **输出**：`UsageAggregate`
- **是否 spawn_blocking**：**是**（聚合计算可能遍历大量记录）
- **是否扫描文件系统**：否（基于已缓存的 UsageRecord）
- **是否适合缓存**：适合（按 `(time_range, group_by)` 缓存 10 秒）
- **错误返回**：Tauri 标准错误

### 6.4 `export_usage_analytics_json`

- **用途**：导出当前 usage 分析结果为 JSON 文件
- **输入**：
  ```rust
  struct ExportUsageAnalyticsRequest {
      time_range: TimeRange,
      group_by: GroupBy,
      output_path: Option<PathBuf>, // None = 使用默认路径（Downloads 或桌面）
  }
  ```
- **输出**：
  ```rust
  struct ExportUsageAnalyticsResponse {
      output_path: PathBuf,
      record_count: usize,
  }
  ```
- **是否 spawn_blocking**：**是**（文件写入）
- **是否扫描文件系统**：写入文件
- **是否适合缓存**：不适用
- **错误返回**：Tauri 标准错误（如目录不可写）

### 6.5 `run_usage_audit_against_ccusage`

- **用途**：运行 ccusage 对账诊断
- **输入**：
  ```rust
  struct RunUsageAuditRequest {
      config_dir: Option<PathBuf>,  // None = 使用所有已识别目录
      ccusage_path: Option<String>, // None = 尝试从 PATH 找 `npx ccusage`
  }
  ```
- **输出**：
  ```rust
  struct UsageAuditReport {
      agent_scope_totals: UsageTotals,
      ccusage_totals: Option<UsageTotals>,
      diffs: Option<UsageDiffs>,
      ccusage_available: bool,
      ccusage_error: Option<String>,
      missing_sessions: Vec<String>,
      extra_sessions: Vec<String>,
  }

  struct UsageTotals {
      session_count: usize,
      input_tokens: u64,
      output_tokens: u64,
      cache_read_tokens: u64,
      cache_create_tokens: u64,
  }

  struct UsageDiffs {
      session_count_diff: i64,
      input_tokens_diff: i64,
      output_tokens_diff: i64,
      cache_read_diff: i64,
      cache_create_diff: i64,
  }
  ```
- **是否 spawn_blocking**：**是**（调用外部 ccusage 命令）
- **是否扫描文件系统**：是（读取 JSONL）
- **是否适合缓存**：不适用（诊断工具）
- **错误返回**：ccusage 不可用时 `ccusage_available = false`，不返回致命错误

## 7. 前端页面设计

### 7.1 页面位置建议

**推荐方案**：在 **"通用监控"（Monitoring）** 域下新增子页面 **"Usage 分析"**

```
通用监控
├── Agent 监控          (现有)
├── 会话管理            (现有)
└── Usage 分析          (新增)
```

理由：

- 不破坏现有项目监控域的导航结构
- Usage Analytics 是跨项目的全局视角，放在"通用监控"合理
- 与 Agent 监控、会话管理形成互补关系

### 7.2 页面布局

```
┌─────────────────────────────────────────────────────────────┐
│  [数据源状态卡片]                                           │
│  来源: Claude JSONL | 3 个目录可读 | 最后更新: 2 分钟前     │
│  [?] 基于已写入 usage，实时速率为估算...                     │
├─────────────────────────────────────────────────────────────┤
```

## 8. Confidence / 可信度文案

### 8.1 核心原则

必须在 UI 中明确传达以下信息，避免用户误解：

1. **数据来源**：基于 Claude Code 已写入的 usage 记录统计
2. **最终可靠性**：最终用量（回复结束后）通常较可靠
3. **实时性限制**：实时速率不是流式精确值
4. **刷新机制**：部分模型或 endpoint 只在回复结束后刷新 usage
5. **成本性质**：成本估算如果后续加入，也只能是估算

### 8.2 建议 Tooltip 文案

**数据源状态卡片问号 Tooltip**：

```
数据来源：Claude Code 本地 usage.jsonl 文件
━━━━━━━━━━━━━━━━━━━━━━
• 统计基于 Claude Code 已写入磁盘的 usage 记录
• 最终用量（对话结束后）通常较可靠
• 实时速率是基于已写入数据的估算，不是流式精确值
• 部分模型或 endpoint 只在回复结束后刷新 usage
• 当前支持：~/.claude、~/.config/claude、CLAUDE_CONFIG_DIR
```

**"等待计数更新"提示**：

```
Claude Code 正在生成回复，usage 计数将在回复结束后更新
当前显示的是上一轮已完成的 usage
```

**速率估算 Tooltip（复用 v0.3.6，微调文案）**：

```
实时速率 = 最近时间窗口内新增 token / 时间差
━━━━━━━━━━━━━━━━━━━━━━
• 1 分钟平均：最近 60 秒内已写入的 token 平均速率
• 5 分钟平均：最近 5 分钟内已写入的 token 平均速率
• 由于 usage 写入有延迟，速率可能存在跳变
• 不代表 API 层的精确 streaming 速率
```

**"延迟更新"标签说明**：

```
延迟更新：usage 数据基于本地文件扫描，可能有秒级到分钟级的延迟
```

### 8.3 页面顶部提示条

建议在 Usage Analytics 页面顶部显示常驻提示：

```
ℹ️ Usage 统计基于 Claude Code 本地已写入的 usage 记录。最终用量较可靠，实时数据可能有延迟。
```

## 9. ccusage 对账诊断设计

### 9.1 目标

- 同一数据目录下，比较 AgentScope parser 和 ccusage 的总量差异
- 用于开发验证，不作为用户主功能
- 输出差异摘要

### 9.2 调用方式

**方式 A：直接调用 npx ccusage**（推荐）

```bash
npx ccusage --json --dir ~/.claude
```

**方式 B：调用已安装的 ccusage**

```bash
ccusage --json --dir ~/.claude
```

### 9.3 是否要求用户安装 ccusage

- **不要求用户安装**
- 如果 ccusage 不存在，`ccusage_available = false`，返回 AgentScope 自身统计
- 诊断脚本提示用户：`"ccusage 未安装，跳过交叉校验。如需安装：npm install -g ccusage"`

### 9.4 对比字段

| 字段 | AgentScope | ccusage | 说明 |
|------|-----------|---------|------|
| session_count | ✓ | ✓ | 会话数量 |
| input_tokens | ✓ | ✓ | 输入 token |
| output_tokens | ✓ | ✓ | 输出 token |
| cache_read_tokens | ✓ | ✓ | cache 读取 |
| cache_create_tokens | ✓ | ✓ | cache 创建 |

### 9.5 容忍误差定义

由于统计口径可能不同（如 session 边界、时间范围、去重策略），定义合理容忍度：

| 指标 | 容忍误差 |
|------|---------|
| session_count | ±0（必须一致） |
| input_tokens | ±1% 或 ±100 tokens |
| output_tokens | ±1% 或 ±100 tokens |
| cache_read_tokens | ±1% 或 ±50 tokens |
| cache_create_tokens | ±1% 或 ±50 tokens |

如果差异在容忍范围内，标记为 `"一致"`，否则标记为 `"差异"`。

## 12. 风险与开放问题

### 12.1 已知风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Claude Code 本地文件结构变化 | 高 | 保持 parser 模块化，便于适配；关注 Claude Code 更新日志 |
| ccusage 统计口径不完全一致 | 中 | 定义容忍误差范围；audit 报告明确标注差异 |
| 多目录重复 session | 中 | 以 `session_id + config_dir` 为 key，按 mtime 去重 |
| 成本估算暂不纳入导致用户期待管理 | 中 | UI 明确标注"v0.3.7 不做成本估算" |
| JSONL 超大文件性能 | 中 | 增量扫描、分段读取、内存缓存 |
| 隐私与本地路径展示 | 低 | 路径展示限制在项目名级别，不暴露完整家目录路径 |
| Windows/macOS/Linux 路径差异 | 中 | 使用 Rust 标准库 Path/PathBuf，平台条件编译 |
| 第三方 Claude-compatible endpoint usage 字段不一致 | 中 | 文档说明当前仅支持 Claude Code 官方 JSONL 格式 |

### 12.2 开放问题

1. **cache token 命名**：Claude Code JSONL 中 `cache_read_input_tokens` vs `cache_creation_input_tokens`，与 Anthropic API 的命名是否完全一致？是否需要兼容旧版本字段名？
2. **模型名称提取**：usage 记录中的模型名来自哪里？是 `message.model` 还是其他字段？需要调研真实 JSONL。
3. **时间戳时区**：JSONL 中的时间戳是 UTC 还是本地时间？前端展示使用本地时区还是 UTC？
4. **增量扫描精度**：如果 Claude Code 在写入过程中文件被截断后重写，基于 offset 的增量扫描是否安全？
5. **与 AgentCollector 的数据一致性**：abtop-collector 实时采集的 usage 与 Usage Scanner 历史扫描的 usage 是否应该合并展示？
