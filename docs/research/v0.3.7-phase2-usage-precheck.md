## 4. session_id 提取

### 是否可从文件名提取

✅ **可以**。文件名格式为 `{session_id}.jsonl`，例如 `1f093b08-04f6-4274-bee7-5dfd63d57d5a.jsonl`。

提取方式：
```rust
let session_id = file_path.file_stem()?.to_str()?.to_string();
```

### 行内是否存在 session id

✅ **存在**。JSONL 每行根级有 `sessionId` 字段，值与文件名一致：

```json
{
  "sessionId": "1f093b08-04f6-4274-bee7-5dfd63d57d5a",
  ...
}
```

### Fallback 建议

- **主策略**：从文件名提取 `session_id`
- **Fallback**：如果文件名不符合 UUID 格式，从行内 `sessionId` 字段提取
- **双保险**：扫描时校验文件名 session_id 与行内 sessionId 是否一致，不一致时记录 warn 日志

## 7. 对 spec 的影响

### 7.1 扫描路径

| 项目 | 结论 |
|------|------|
| 主扫描路径 | `{config_dir}/projects/**/*.jsonl` ✅ 正确 |
| usage.jsonl | 本机不存在，但保留兼容路径是合理的 |
| 项目目录名 | 是编码后的绝对路径，非人类可读项目名 |

**Phase 2 编码注意事项**：
- 项目目录名是编码路径（如 `-Users-ckstar-Repo-agent-scope`），不是简短项目名
- 需要解码或映射到 AgentScope `ProjectRegistry` 中的项目名
- 可考虑将编码路径与 `ProjectRegistry` 中的 `project_path` 进行匹配

### 7.2 Parser 字段

| 项目 | 结论 |
|------|------|
| usage 路径 | `.message.usage` ✅ 已确认 |
| model 路径 | `.message.model` ✅ 已确认 |

| sessionId 路径 | 根级 `.sessionId` ✅ 已确认 |

**Phase 2 编码注意事项**：
- parser 应从 `.message.usage` 提取 token 字段
- 如果 `.message.usage` 为 `null`，跳过该行（这是正常的，表示 usage 尚未写入）
- model 可能为 `null`，需处理为 `Option<String>`

### 7.3 ccusage 对账命令

| 项目 | 结论 |
|------|------|
| ccusage 命令 | `npx ccusage daily --json` |
| `--dir` 参数 | ❌ 不支持 |
| 自动发现 | ✅ ccusage 自动发现配置目录 |

**Phase 6 编码注意事项**：
- audit 命令应调用 `npx ccusage daily --json`（或对应子命令）
- 不支持 `--dir`，所以 audit 只能对比 ccusage 自动发现的目录
- 如果需要限定目录，可能需要通过环境变量或 ccusage 配置文件实现
- 对账时注意 camelCase ↔ snake_case 转换

### 7.4 Phase 2 编码总体建议

1. **扫描器**：实现 `{config_dir}/projects/**/*.jsonl` 的 glob 扫描
2. **解析器**：
   - 解析 `type == "assistant"` 的行
   - 从 `.message.usage` 提取 4 个 token 字段
   - 处理 `usage == null` 的情况
   - 从文件名提取 `session_id`
   - 从 `.message.model` 提取模型名
   - 从 `.timestamp` 提取时间戳
3. **项目映射**：考虑编码目录名如何映射到人类可读项目名
4. **增量扫描**：JSONL 文件是追加写入的，适合基于 offset 的增量扫描
